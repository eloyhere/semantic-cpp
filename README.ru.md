# 🚀 Semantic-Cpp: современный интеллектуальный фреймворк потоковой обработки для будущего

Semantic-Cpp — это полностью переработанная современная библиотека потоковой обработки на C++, использующая модульную архитектуру **«множество заголовочных файлов, нулевые внешние зависимости»**. Каждый заголовочный файл имеет чёткую ответственность, независим и тестируем, формируя вместе целостную экосистему потоковой обработки. Библиотека творчески объединяет лучшие черты различных парадигм программирования:

- 🎯 Элегантность и плавность **Java Stream API**: цепочечные вызовы, декларативное программирование, код словно стихи.
- ⚡ Ленивость и гибкость **JavaScript Generator**: отложенные вычисления, генерация по требованию, эффективное использование памяти.
- 🗄️ Эффективность и упорядоченность **индексов баз данных**: умная сортировка, управление через индексы, идеально для временных рядов.

В отличие от традиционной обработки данных (ручные циклы, асинхронные колбэки), Semantic-Cpp предоставляет **безопасное по типам, выразительное и высокопроизводительное** решение. Его ключевая философия проектирования — **точное управление потоком данных**: данные текут только тогда, когда это необходимо, а порядок и позиция точно регулируются **«индексами»** для оптимального использования ресурсов.

## 📐 Архитектура проекта: семислойный модульный дизайн

Semantic-Cpp состоит из **семи основных заголовочных файлов**, каждый из которых выполняет свою независимую роль:

```
┌─────────────────────────────────────────────────┐
│                 semantics.h                     │
│    (Фабрики потоков: числовые диапазоны,        │
│     контейнеры, текст, Unicode)                  │
├─────────────────────────────────────────────────┤
│                  semantic.h                     │
│   (Промежуточные операции, система Collectable, │
│    специализации контейнеров)                   │
├─────────────────────────────────────────────────┤
│                 collectors.h                    │
│  (Фабрики коллекторов: сопоставление, поиск,    │
│   агрегация, статистика, DFT/FFT и др.)          │
├─────────────────────────────────────────────────┤
│                 collector.h                     │
│     (Каркас коллекторов: пятиэтапная модель,    │
│      поддержка параллелизма и конкуренции)       │
├─────────────────────────────────────────────────┤
│                charsequence.h                   │
│  (Последовательности Unicode, мультикодировка,  │
│   Builder, Buffer)                              │
├─────────────────────────────────────────────────┤
│                   pool.h                        │
│   (Глобальный пул потоков: задачи, аварийное    │
│    завершение, распространение исключений)       │
├─────────────────────────────────────────────────┤
│                 function.h                      │
│  (Определения типов: Generator, Supplier,       │
│   Consumer и др. псевдонимы)                     │
└─────────────────────────────────────────────────┘
```

### Граф зависимостей

```
function.h          ← без зависимостей, краеугольный камень типов
pool.h              ← зависит от function.h
charsequence.h      ← независимый модуль, работа с Unicode
collector.h         ← зависит от function.h, pool.h
collectors.h        ← зависит от collector.h, charsequence.h
semantic.h          ← зависит от collector.h, collectors.h, charsequence.h
semantics.h         ← зависит от semantic.h
```

Каждый заголовочный файл можно компилировать и тестировать независимо или подключать по мере необходимости.  
Например, если нужны только коллекторы, достаточно включить `collector.h` и `collectors.h`.

## 🏗️ Слой 1: function.h — основа типов

`function.h` определяет систему типов всей библиотеки — общий фундамент всех модулей:

```cpp
namespace function {
    using Timestamp = long long;           // тип индекса
    using Module = unsigned long long;     // тип модуля / счётчика
    
    template <typename T>
    using Generator = std::function<void(
        std::function<void(T, Timestamp)>,      // accept
        std::function<bool(T, Timestamp)>       // interrupt
    )>;
}
```

`Generator` — центральная абстракция всей системы потоков: принимает два колбэка — `accept` (приём данных) и `interrupt` (прерывание обработки) — воплощая модель **«ленивого вытягивания» (lazy pull)**.

## 🔧 Слой 4: collector.h — каркас коллекторов

`collector.h` реализует паттерн Collector, являющийся ядром терминальных операций.

### Пятиэтапная модель

```
Identity → Accumulator → Combiner → Finisher
              ↑
           Interrupt (опциональное короткое замыкание)
```

### Псевдонимы типов

| Тип | Определение | Роль |
|-----|------------|------|
| `Identity<A>` | `Supplier<A>` | Предоставляет начальное значение |
| `Accumulator<A,E>` | `TriFunction<A,E,Timestamp,A>` | Накопление элементов |
| `Combiner<A>` | `BiFunction<A,A,A>` | Объединение параллельных результатов |
| `Finisher<A,R>` | `Function<A,R>` | Финальное преобразование |
| `Interrupt<E,A>` | `TriPredicate<E,Timestamp,A>` | Короткое замыкание |

### Поддержка конкурентности

`Collector::collect()` автоматически обрабатывает:

- 📦 Шардирование данных (распределение по потокам по модулю индекса)
- 🔗 Слияние результатов (через `Combiner`)
- ⚠️ Распространение исключений (через `std::exception_ptr` и `std::atomic<bool>`)

## 🧠 Ключевая концепция: мир данных, управляемый индексами

Semantic-Cpp абстрагирует обработку данных как операции над **«элементами»** и их **«логическими позициями (индексами)»**. Понимание этого — ключ к овладению библиотекой.

### 1. 📐 Базовые преобразования индексов

| Метод | Описание |
|------|----------|
| `redirect(fn)` | Ядро: полная перезапись индексов через пользовательскую функцию |
| `reverse()` | Инверсия всей логики индексов (через `redirect`) |
| `translate(offset)` | Фиксированное смещение |
| `translate(translator)` | Динамическое смещение по элементу и индексу |

### 2. 📊 «Авторитарное» правило сортировки

> ⚠️ **`sort()` переопределяет всё**: после вызова все предыдущие операции с индексами отбрасываются; элементы получают новые естественные индексы на основе значений.

- `sort()` → немедленно материализуется в `OrderedCollectable`, сортировка по значению
- `sort(comparator)` → сортировка по пользовательскому компаратору

### 3. ⚡ Декларативная параллельная обработка

- `parallel(n)` только объявляет намерение; потоки не запускаются сразу
- Терминальные операции (`toUnordered()`, `count()` и др.) запускают параллелизм
- Пул потоков автоматически распределяет задачи и объединяет результаты

### 4. 🎯 Выбор итогового контейнера

| Метод преобразования | Внутренняя структура | Характеристики производительности | Лучший сценарий |
|---------------------|---------------------|-----------------------------------|----------------|
| `sort()` | `OrderedCollectable` | Материализация и сортировка по значению | Сортировка, пагинация, временные ряды |
| `toOrdered()` | `OrderedCollectable` | Сохранение текущего порядка индексов | Сохранение пользовательского порядка |
| `toUnordered()` | `UnorderedCollectable` | O(1) в среднем, максимальная производительность | Быстрый поиск, дедупликация, агрегация |
| `toWindow()` | `WindowCollectable` | На основе упорядоченных коллекций | Скользящие/перекатывающиеся окна |
| `toStatistics()` | `Statistics` | 20+ статистических методов | Комплексный статистический анализ |

    .map(int x { return x * x; })

## ⚡ Рекомендации по оптимизации производительности

1. 🎯 **Выбирайте правильный контейнер**
   - Поиск по равенству, несортированная агрегация → `toUnordered()`
   - Диапазонные запросы, сортировка, пагинация → `toOrdered()` или `sort()`
   - Анализ окон реального времени → `toWindow()`
2. ⚡ **Разумно используйте параллелизм**: для больших данных или тяжёлых вычислений используйте `parallel()`, избегайте блокирующего ввода-вывода.
3. 📐 **Оптимизируйте порядок операций**: фильтруйте раньше, сортируйте осознанно.
4. 🔄 **Используйте ленивость**: промежуточные операции выполняются только при терминальной операции; `takeWhile` и `limit` позволяют досрочно завершить обработку.
